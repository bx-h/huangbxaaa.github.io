    摘要：String; StringBuffer; StringBuilder; 包装类

    String #

    1. 字符串属于引用数据类型。
    2. 使用双引号括起来的都是String对象。
    3. String对象一旦出生不可变。
    4. 垃圾回收器不会释放常量。
    5. 凡是用双引号括起来的字符串都是存储在“方法区”的“字符串常量池”中。

    public class StringTest {
    public static void main(String[] args) {
        String x = new String("xyz");
        String y = new String("xyz");
        System.out.println(x == y);
        // 输出是？？？
        // false
    }
}

    原因：会在堆中创建两个不同的对象指向字符串常量池中同一个“xyz”。所以x和y指向的堆中的对象不同，地址也就不同。

    因此，比较String要调用equals方法。

    StringBuffer #

    由于字符串的不可变，字符串的频繁拼接会占用大量方法区内存，造成内存空间的浪费。

    因此，需要采用新的类型：StringBuffer。

    实现与方法 #

    底层是byte[]。默认初始容量为16。

    注意：String底层是final byte[]，指向数组的局部变量的指向无法改变，因此进行修改时，老数组无法被释放

    注意：数组的容量确定之后，不能改变。

    append()：追加。当byte[]数组满了之后，会创建更大的数组，进行arraycopy，并改变指向数组的局部变量的指向，而原byte数组就会被垃圾回收期回收。

    如何优化StringBuffer的性能？

    在创建StringBuffer时给定一个稍大的初始化容量，减少扩容次数。

    同样可以进行追加。

    StringBuilder与StringBuffer的区别 #

    StringBuffer中的方法都有：synchronized关键字修饰。在多线程环境下运行是安全的。

    StringBuilder中的方法没有：synchronized关键字修饰。他不是线程安全的。

    class TestInteger{
	public static void main(String[] args) {
		Integer x1 = 1000;
		Integer x2 = 1000;
		System.out.println(x1 == x2); // false
        
        Integer y1 = 127;
        Integer y2 = 127;
        System.out.println(y1 == y2); // true
	}
}

    原因：Java中为了提高程序的执行效率，在Integer类加载时，会将[-128, 127]之间所有的包装对象提前创建好，放到了方法区的“整数型常量池”当中。因此使用这些数字时不需要new，直接指向常量池中的对应数字。
